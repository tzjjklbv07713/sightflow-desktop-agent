# SightFlow 长期项目重整规划

## 目标

把当前已经能工作的桌面自动化能力，拆成可持续演进的四层：

1. 感知层：截图、窗口、红点、布局、未读检测。
2. 决策层：回复模型、视觉模型、Provider、运行时状态。
3. 执行层：点击、输入、粘贴、发送、切换会话。
4. 产品层：设置页、启动流程、文档、长期演进约束。

当前第一阶段不追求重写，而是先把职责边界固定住，避免后续每加一个能力都把视觉、回复、发送、状态混成一团。

## 现状判断

现在项目已经有几个明显的分界面，但边界还不够硬：

- `src/core/ai-client.ts` 同时承担聊天回复和 VLM 调用。
- `src/core/rpa/vision-utils.ts` 既做布局检测，也反向推输入框，还直接依赖 AI 客户端。
- `src/core/rpa/input-utils.ts` 已经是执行层的稳定入口，适合继续往下沉，而不是被上层逻辑污染。
- `src/main/index.ts` 把设置、Provider 安装、运行时启动、模型测试都串在一起，适合保留为编排层，但不适合继续增长业务细节。

## 第一阶段

当前实施状态：

- 已拆出 `reply-client`、`vision-client` 和 `model-clients`，旧 `ai-client` 保留兼容外壳。
- 已把内置回复 Provider 改为使用独立的 `replyModel` 配置。
- 已保留 `vision` 配置给布局、未读、坐标类视觉识别。
- 已把输出动作默认改为 `typing-with-paste-fallback`，并支持输入速度配置。
- 已修正当前会话变化检测：每轮 diff 使用当前聊天区截图，并把该截图复用给后续观察，避免旧截图导致漏检。
- 已把未读红点轮询从全窗口截图后裁剪，改为优先局部截图，降低长期运行时的截图开销。
- 已加入轻量性能指标日志，覆盖布局测量、截图、diff、未读检测、回复服务和发送动作耗时。
- 已加入空闲轮询退避：连续无新消息时从 5 秒逐步放慢到最多 15 秒，检测到变化或完成回复后恢复快速轮询。
- 已在主界面聚合性能指标，展示最近耗时和平均耗时，并支持按日志类型筛选。
- 已把最新消息来源观察结果传入回复 Provider，降低回复模型误读历史消息或自我消息的概率。

### 1. 拆模型职责

把模型调用拆成两个明确方向：

- 视觉模型：只负责截图理解、布局/未读/坐标类输出。
- 回复模型：只负责根据聊天截图生成回复文本。

保留一个兼容壳，避免一次性改掉所有消费点，但新代码必须走明确职责的入口。

### 2. 固化执行层

把“发送消息”当成执行动作，不让它再承担决策逻辑。

- 输入动作只接受已决定的文本。
- 输出模式只处理 `typing` / `paste` / `typing-with-paste-fallback`。
- 后续所有“怎么输入”的变化，尽量只改执行层。

### 3. 稳定状态边界

把设置、运行配置、设备运行时状态明确区分：

- 设置层：用户持久化配置。
- 运行配置层：启动时注入到设备和 Provider 的有效配置。
- 运行时层：当前窗口、缓存、布局、节流、会话状态。

### 4. 降低截图成本

长期目标不是“时刻截图”，而是把截图变成有条件触发：

- 红点识别优先用局部区域或缓存。
- 布局测量尽量复用缓存。
- 对话变化尽量先做差分或状态判断，再决定是否调用视觉模型。

## 第二阶段

### 1. 事件化检测

把“轮询截图”改成“状态驱动 + 必要时截图”。

优先顺序建议是：

1. 未读标记变化。
2. 会话切换或点击动作后窗口状态变化。
3. 聊天区域 diff。
4. 最后才是完整截图 + VLM。

### 2. 回复链路分层

回复链路建议拆成三段：

- 观察：当前看到什么。
- 决策：该不该回、回什么。
- 执行：如何输入并发送。

这样后面你要加“主人风格学习”“逐字输入”“人工接管”“回退策略”时，不会污染模型层。

当前已完成第一步：`replyContext.latestMessage` 会随 Provider 输入传入回复链路，内置 Provider 会把它写入模型提示词。下一步可以继续补 OCR 或消息块裁剪，把“最新左侧消息文本”从整张截图里提取出来，再交给回复模型。

### 3. 模型能力分类

不要把“多模态模型”默认等同于“视觉模型”。

更稳妥的做法是把能力显式分类：

- `vision`：读图、框选、布局、未读检测。
- `chat`：回复内容生成。
- `ocr`：如果以后单独引入文字提取。
- `routing`：如果以后引入多模型选择或中转站。

一个模型可以具备多个能力，但项目代码里不要把它们默认混成同一类。

## 第三阶段

### 1. 学习与个性化

可以做“自我进化”，但建议严格限定在行为适配层，不直接让模型自发改写策略：

- 记录主人对不同语气、场景、对象的最终回复。
- 提取可控特征，如长短、语气、表情、称呼、是否追问。
- 形成可解释的回复偏好，而不是黑箱训练。

### 2. 人工可控回退

所有自动化都要留回退口：

- 视觉失败回退到缓存或手动框选。
- 回复失败回退到粘贴。
- 逐字输入失败回退到粘贴。
- 模型超时回退到跳过或稍后重试。

### 3. 可靠性监控

长期项目必须补三类观测：

- 成功率。
- 响应耗时。
- 回退次数。

没有这三项，后期很难判断是识别坏了、回复坏了，还是执行坏了。

## 推荐的迭代顺序

1. 先固定模型分层。
2. 再把执行层和设置层边界收紧。
3. 然后做事件驱动和减少截图。
4. 最后再上学习能力和更强的自动优化。

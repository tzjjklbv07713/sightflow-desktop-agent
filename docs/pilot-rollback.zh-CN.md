# SightFlow 商用试点回滚方案

当试点环境出现异常、或者需要把客服电脑从 SightFlow 模式恢复为"完全人工"模式时,按本文档操作。

## 1. 触发回滚的常见信号

在以下情况出现任意一条,都应该立刻触发回滚而不是继续运行:

1. Console 出现连续 3 条以上 failed 记录,且失败原因都是 send-failed 或 input-not-found。
2. Console 出现 circuit-breaker-open 状态,即便点重置也立刻再次熔断。
3. 同一客户在 5 分钟内被自动回复超过 3 次(重复回复去重失效)。
4. 视觉模型或回复模型服务长时间不可达,且本地缓存也没有命中。
5. 客服人员明确反馈"不该回的回了"或者"该回的没回"。
6. Windows 微信 / 企微更新后界面布局发生较大变化,识别准确率显著下降。

## 2. 回滚等级

SightFlow 提供三级回滚,按影响面从小到大:

### 等级 A:暂停自动发送(秒级生效,不卸载)

适用:个别会话出错,或者需要临时人工接管。

操作:

1. 在主界面点"停止引擎",或者在 Console →「会话记录」里把出错会话设为"人工接管"。
2. 引擎会停止扫描新消息,但保留所有已识别的草稿和会话记录。
3. 恢复时点"启动引擎",或者把会话从"人工接管"切回"自动"。

回滚后状态会保留:设置、知识库、授权、所有 Console 记录。

### 等级 B:关闭自动启动 + 保留安装(分钟级生效)

适用:试点阶段发现严重策略问题,但还需要导出诊断和会话记录给工程团队复盘。

操作:
1. 等级 A 操作做完后,关闭 SightFlow 主进程。
2. 打开任务管理器,确认 SightFlow.exe 已经退出。
3. 在开始菜单找到"SightFlow",右键 → 打开文件位置,把启动快捷方式从"启动"文件夹移除(避免下次开机自启)。
4. 导出诊断包:Console →「诊断」面板 → 导出诊断包,把 .zip 文件发给工程。
5. 知识库和授权文件保留在 %APPDATA%\SightFlow\,供后续调查。

回滚后状态保留:所有设置、知识库、授权、Console 记录、诊断包。SightFlow 程序本身保留,客服可以在排查完成后重新启动。

### 等级 C:完全卸载(不可逆)

适用:SightFlow 试用决定不继续,或者客户机器要重装系统。

操作:
1. 先在 Console →「诊断」面板导出诊断包,作为留档。
2. 在 Windows 设置 → 应用 → 已安装的应用里找到 SightFlow,点卸载。
3. 安装器自带的 NSIS 卸载器会移除:程序文件、桌面快捷方式、开始菜单项、注册表项。
4. 手动删除残留的用户数据(可选,只在你确定不再用时执行):
   - %APPDATA%\SightFlow\         用户数据、设置、授权、Console 记录
   - %LOCALAPPDATA%\SightFlow\    日志、缓存、截图
5. 重启电脑,确认 SightFlow.exe 已经完全消失。

卸载完成后,客户机器等同于从未安装过 SightFlow,Windows 微信 / 企微 不受影响。

## 3. 关键文件位置参考

| 类别 | 路径 | 说明 |
| --- | --- | --- |
| 程序 | C:\Program Files\SightFlow\ | 安装器部署位置 |
| 用户数据 | %APPDATA%\SightFlow\settings.json | 当前设置 |
| 用户数据 | %APPDATA%\SightFlow\license.json | 授权状态 |
| 用户数据 | %APPDATA%\SightFlow\onboarding.json | 初始化完成状态 |
| 用户数据 | %APPDATA%\SightFlow\traces\ | 会话审计 trace |
| 用户数据 | %APPDATA%\SightFlow\screenshots\ | 识别证据截图 |
| 日志 | %LOCALAPPDATA%\SightFlow\logs\ | 主进程日志、渲染进程日志 |
| 知识库 | %APPDATA%\SightFlow\knowledge\ | 导入的知识库文件 + 索引 |
| 诊断包 | 用户选择位置 | Console 导出的 .zip 诊断包 |

等级 B 之前,建议先把这些目录打包留档。等级 C 卸载时只会清程序,不会动这些用户数据,需要手动删。

## 4. 回滚后的复盘动作

完成等级 B 或 C 回滚后,请按顺序做以下动作:

1. 把诊断包附在工单里,工单模板请参考内部 wiki 的"试点工单模板"。
2. 在工单里写清:回滚等级、触发原因、首次发现时间、影响客户数、最后一条成功的 trace id。
3. 工程团队会在 24 小时内给出:是否需要在工程环境复现、是否需要修复发版、是否要追加新的冒烟用例。
4. 修复后的 SightFlow 版本,先在工程环境验证,然后给试点客户的 1 台机器升级,观察 24 小时后再批量升级。
5. 任何等级 C 回滚都需要商务侧同意,不要自行卸载客户机器。

## 5. 回滚期间客户须知

回滚期间,客户侧的体感变化:

- 等级 A:完全不感知。SightFlow 仍在运行,但不再自动发送消息,客服需手动回复。
- 等级 B:不再弹窗打扰,但 SightFlow 图标仍在任务栏。客服可正常关闭,知识库和记录都在,排查完后能直接重启。
- 等级 C:完全移除 SightFlow,客服回到 100% 人工状态。所有历史数据留在工程侧工单里。

回滚期间请告知客服:不要卸载或重装 微信 / 企微,否则会丢失会话上下文。
